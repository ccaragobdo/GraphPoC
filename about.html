BDO Data Governance Assessment

Data Architecture Scanner — About & Step Reference

All steps authenticate with a single Microsoft Graph delegated access token. Each step can be run independently. Feasibility ratings reflect what is accessible from the current Graph/SharePoint scanner path vs. what requires Purview or compliance-plane access.

1 Token Setup Required for all steps

Paste a Microsoft Graph delegated access token obtained from Graph Explorer or Azure CLI/PowerShell. The token must have the following scopes:

  • User.Read — identity verification
  • Files.Read.All — read file metadata across drives
  • Sites.Read.All — enumerate SharePoint sites and libraries
Token lifetime is typically 1 hour. If you see 401 errors, regenerate a fresh token from Graph Explorer.

2 Content Staleness Scan Fully feasible via Graph

Discovers all accessible SharePoint sites and then samples files within each library to measure content age. Uses time and file-count caps to stay within practical run times.

  • Site discovery via /sites/getAllSites with fallback to /sites?$top=999, /me/followedSites, and /sites?search=*
  • File records collected: name, extension, size, created date, last modified date
  • Staleness calculated at 90 / 180 / 365 day thresholds
  • Results aggregated per site and per library
  • Configurable: max sites, max files, max runtime, folder depth, OneDrive inclusion

Exports: summary_by_site.csv, summary_site_exposure.csv, full_results.json

This is a time-bounded sampling assessment, not a full inventory. Large tenants will be partially sampled.

3 Sharing & Permissions Exposure Partially feasible — tenant policy dependent

Re-discovers all accessible SharePoint sites and checks sharing permissions at the site level, library root level, and top-level folder level. Does not require Step 2 to have been run first.

  • Anonymous / Anyone links — sharing links with scope: anonymous
  • Organization-scoped links — links accessible to all tenant members
  • Everyone / All Users grants — direct permission grants to broad groups
  • Checked at: site permissions endpoint, library root, and top-level folders
  • 403 access-denied responses are silently skipped (tenant policy controls this)

Exports: step_3_results.json, step_3_results.csv (per-site exposure rows)

The /sites/{id}/permissions endpoint returns 403 in many tenants for delegated tokens even with Sites.Read.All. Library and folder permission checks are attempted regardless.

4 Sensitivity Labels & Purview Integration Partial — label catalog only via Graph

Attempts to retrieve the tenant sensitivity label catalog via Microsoft Graph to inventory what classification labels are configured.

  • Calls /security/informationProtection/sensitivityLabels
  • Returns label names, IDs, and priority ranking if accessible
  • Confirms whether a label taxonomy exists (Public / General / Confidential / Highly Confidential)

Exports: step_4_results.json, step_4_results.csv (label catalog)

Not available via this path: file-level applied-label coverage, percentage of files labeled, auto-label vs. manual attribution, label inheritance, and auto-labeling policy configuration. These require the Purview Information Protection compliance-plane APIs (separate authentication and licensing).

5 Sensitive Information Types (SITs) Not feasible via current Graph path

SIT detection identifies files containing regulated data patterns such as Social Security Numbers, credit card numbers, financial data, PII, and healthcare identifiers. This step documents the gap.

  • Microsoft Graph file metadata endpoints do not expose SIT detection results
  • SIT hits are generated by the Purview DLP classification engine and stored in the compliance telemetry pipeline
  • Overshared locations containing SITs cannot be identified without compliance plane access
  • SIT policy match counts and enforcement outcomes are also inaccessible from this path
Requires: Purview DLP/classification telemetry APIs, Microsoft 365 E5 Compliance or equivalent licensing, and a service principal with Compliance Administrator delegated access.

6 Data Loss Prevention (DLP) Not feasible via current Graph path

DLP assessment measures policy coverage (SharePoint, OneDrive, Teams), enforcement mode (audit-only vs. block vs. override with justification), and incident trend volume. This step documents the gap.

  • DLP policy inventory is not exposed through the Microsoft Graph /sites or /drives path
  • DLP incident data lives in the Security & Compliance Center, not in Graph file metadata
  • Policy actions (block / notify / justify) require the Purview DLP policy management API
Requires: Purview DLP APIs (/beta/security/dataLossPreventionPolicies), Microsoft 365 E3/E5 Compliance licensing, and Compliance Administrator access scope.

7 SharePoint Information Architecture Fully feasible via Graph

Assesses the structural quality of each SharePoint library for search effectiveness and Copilot grounding. Walks every discovered site and library to collect structural signals.

  • Folder-only pattern — libraries where the root contains only folders and no loose files (anti-pattern)
  • Deep nesting — measures actual folder depth; flags libraries exceeding 5 levels
  • Custom columns — counts non-hidden, non-read-only columns per library (metadata richness indicator)
  • Multiple content types — detects libraries with more than one content type (structured taxonomy signal)
  • Powered by: /drives/{id}/root/children, /sites/{id}/lists/{id}/columns, /sites/{id}/lists/{id}/contentTypes

Exports: step_7_results.json, step_7_results.csv (per-library row)

Term store managed metadata columns and taxonomy consistency across sites require additional calls to the SharePoint REST API (/_api/site/rootweb/termstore), which is outside this scanner's Graph-only path.

8 Search Readiness Signals Feasible via Graph (heuristic)

Samples files across discovered sites and applies heuristic rules to flag content that degrades Microsoft Search and Copilot grounding quality.

  • Generic / untitled file names — flags patterns like Document1.docx, Copy of..., Scan001.pdf, File.xlsx
  • Duplicate file names — names appearing more than once across the sampled set
  • Potentially non-indexed types — extensions unsupported by Microsoft Search indexing (.exe, .pst, .iso, .bak, .zip, etc.)
  • Configurable sample size (default: 2,000 files per run)

Exports: step_8_results.json, step_8_results.csv (per-file flags)

Graph does not reliably expose SharePoint list-item field values (Title, Description, custom metadata) without expensive per-item expansion. Metadata completeness signals are therefore heuristic only.

9 Lifecycle & Records Management Partial — site staleness feasible; retention requires Purview

Measures site-level activity age and flags sites that have not been modified within a configurable threshold. Identifies orphaned or dormant sites as data lifecycle risk.

  • Site last-modified date — pulled from /sites/{id} metadata
  • Stale site flag — configurable threshold (default: 180 days)
  • Stale site count and percentage reported across all discovered sites

Exports: step_9_results.json, step_9_results.csv (per-site staleness row)

Not available via this path: retention policy configuration and coverage percentage, records declared vs. non-records, version history depth per file, draft vs. published content, and Teams/Viva Engage inactivity signals. These require Purview Records Management APIs and Teams reporting telemetry.
